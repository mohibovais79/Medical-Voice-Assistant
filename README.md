# Medical Voice Assistant — Patient Registration via Voice AI

A voice-based AI agent that answers a real phone number, conversationally
collects U.S. patient demographic information, persists it to a database, and
exposes it through a REST API. Built as a take-home technical assessment for a
Voice AI / Conversational AI Engineer role.

## Architecture

```
Caller ──phone──> Vapi ──webhook──> FastAPI (Render)
                      │                 │
                      │  Groq LLM ◄─────┤  (tool-calling for field
                      │                 │   extraction + validation)
                      │                 │
                      │                 ▼
                      │            Supabase (Postgres)
                      │                 │
                      └── REST API ◄────┘  (GET/POST/PUT/DELETE /patients)
```

**Flow:**
1. A caller dials the Vapi-provisioned phone number.
2. Vapi answers, streams audio through Deepgram (STT) and ElevenLabs (TTS),
   and routes the LLM conversation to **Groq** (Llama 3.3 70B) using our
   system prompt.
3. The LLM collects patient demographics one field at a time, naturally.
   When it has all required fields, it reads them back and asks for
   confirmation.
4. On confirmation, the LLM invokes the `save_patient` tool → Vapi sends a
   `tool-call` webhook to our FastAPI server → we validate + persist to
   Supabase Postgres → return the result → the LLM confirms to the caller.
5. The REST API (`/patients`) allows querying / creating / updating /
   soft-deleting patient records independently of the voice flow.

## Tech Stack & Justification

| Layer | Choice | Why |
|---|---|---|
| **Telephony + Voice** | Vapi | Abstracts STT/TTS/telephony; generous free credits; fastest path to a working dialable number. Official Python SDK (`vapi_server_sdk`) for management. |
| **LLM** | Groq (Llama 3.3 70B Versatile) | Free tier, ultra-low latency (~300ms token time) — critical for natural voice conversations. Strong tool-calling support. |
| **Backend** | Python FastAPI | Async, fast to build, auto-generated Swagger docs, excellent Pydantic validation. |
| **Database** | Supabase (Postgres) | Managed Postgres accessed via direct connection (SQLModel + Alembic migrations). Survives restarts, proper constraints + soft-delete. |
| **Hosting** | Render | Free tier, simple Python deploys, auto-sleep on free tier is acceptable for a demo. |

## Setup

### Prerequisites

- Python 3.11+
- Accounts + API keys for: **Supabase**, **Groq**, **Vapi**
- (Optional) A Render account for hosting

### 1. Clone & install

```bash
git clone https://github.com/mohibovais79/Medical-Voice-Assistant.git
cd Medical-Voice-Assistant
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
pip install -r requirements-dev.txt  # for tests
```

### 2. Configure environment

```bash
cp .env.example .env
# Edit .env with your real keys:
#   SUPABASE_CONN   — from Supabase Dashboard > Project Settings > Database >
#                     Connection string > Transaction pooler > URI
#   GROQ_API_KEY    — from https://console.groq.com/keys
#   VAPI_API_KEY    — from https://dashboard.vapi.ai
#   VAPI_SERVER_URL — your public URL (Render URL or ngrok URL)
```

### 3. Initialize the database (Alembic migration)

The schema is managed by Alembic. Run the migration to create the `patients`
table with all constraints, indexes, the `updated_at` trigger, and 2 seed
patients:

```bash
uv run alembic upgrade head
```

### 4. Run locally

```bash
uvicorn app.main:app --reload --port 8000
# API docs at http://localhost:8000/docs
```

For Vapi to reach your local server, expose it with ngrok:

```bash
ngrok http 8000
# Copy the https URL and set VAPI_SERVER_URL in .env to that URL
```

### 5. Create the Vapi assistant + provision a phone number

```bash
# Create the assistant (uses the system prompt + tools from app/voice/)
python -m app.voice.assistant create-assistant
# Note the "id" from the output
```

Then provision a phone number via the **Vapi dashboard**:

1. Go to https://dashboard.vapi.ai > **Phone Numbers** > **Buy Number**.
2. Pick a US area code and purchase the number.
3. Attach the assistant you just created to that phone number
   (select it in the "Assistant" dropdown).
4. The number is now dialable — callers will reach your assistant.

> Phone number provisioning is done in the Vapi dashboard (not via code)
> to keep the setup simple and avoid accidental charges.

### 6. Deploy to Render

1. Push to GitHub.
2. In Render: **New > Web Service > connect your repo** (or use `render.yaml`).
3. Set the env vars (same as `.env`) in the Render dashboard.
4. Deploy. Render will run `uvicorn app.main:app --host 0.0.0.0 --port $PORT`.
5. Update `VAPI_SERVER_URL` in Vapi (via `update-assistant`) to the Render URL.

### 7. Test

```bash
# Run unit + integration tests
pytest tests/ -v

# Test the API directly
curl http://localhost:8000/patients
curl -X POST http://localhost:8000/patients \
  -H "Content-Type: application/json" \
  -d '{"first_name":"Test","last_name":"User","date_of_birth":"1990-01-15",
       "sex":"Male","phone_number":"5551234567","address_line_1":"1 Test St",
       "city":"Testville","state":"CA","zip_code":"90001"}'

# Call the phone number and register a patient!
```

## API Reference

All responses use the envelope: `{"data": <payload|null>, "error": <string|null>}`

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/patients` | List all (non-deleted) patients. Filters: `?last_name=`, `?date_of_birth=`, `?phone_number=` |
| `GET` | `/patients/:id` | Get one patient by UUID |
| `POST` | `/patients` | Create a patient. 409 if phone already exists. |
| `PUT` | `/patients/:id` | Partial update. Only non-null fields are applied. |
| `DELETE` | `/patients/:id` | Soft-delete (sets `deleted_at`). |
| `GET` | `/health` | Health check |
| `GET` | `/docs` | Swagger UI |

## Voice Agent Design

The system prompt (`app/voice/prompt.py`) is the most important artifact for
conversational quality. Key design decisions:

- **One question at a time** but accepts out-of-order answers if the caller
  volunteers extra info.
- **Natural interpretation** — "the fourth of July nineteen ninety" → `07/04/1990`.
- **Spell-out confirmation** for names: "S-M-I-T-H, right?"
- **Mandatory read-back** before saving — the LLM must never call `save_patient`
  without explicit caller confirmation.
- **Field-specific re-prompts** — when a tool returns a validation error, the
  LLM re-asks for that field in plain English (never reads the raw error).
- **Duplicate detection** — when the caller gives their phone, the LLM calls
  `lookup_patient_by_phone` first and offers to update if a record exists.
- **Brevity** — max 2-3 sentences per response. This is a phone call.

### Tools (function-calling)

| Tool | When | What it does |
|---|---|---|
| `lookup_patient_by_phone` | After caller gives phone | Checks DB for existing patient → duplicate detection |
| `save_patient` | After confirmation | Validates + inserts a new patient record |
| `update_patient` | Returning caller confirms changes | Validates + partial-updates an existing record |

## Observability

- All tool calls, DB writes, and webhook events are logged to stdout
  (Render captures stdout).
- Call transcripts are logged on `end-of-call-report` (bonus: persisting to
  `call_transcripts` table is stubbed but not wired in the 3-hour version).

## Known Limitations & Trade-offs

- **No webhook signature verification** — Vapi can sign webhooks with a public
  key; we accept all POSTs for simplicity. In production, verify the signature.
- **CORS is wide open** (`*`) — fine for a demo; tighten to known origins in prod.
- **Service-role Supabase key** — used server-side only (never exposed to
  browser), but in production you'd use Row Level Security + anon key + JWT.
- **No HIPAA compliance** — this is a technical assessment, not a production
  healthcare system. Do not store real patient data.
- **Render free tier sleeps** — the first call after idle may take ~30s to wake.
  For a always-on demo, upgrade to a paid plan.
- **Call transcripts** are logged but not persisted to the DB table in this
  version (stubbed in `webhook.py`).
- **Single-language** — the prompt is English-only. Multi-language (Spanish)
  is a documented bonus; the prompt mentions it but Groq tool-calling in
  Spanish is not tested.
- **No retry on DB write failure** — the LLM tells the caller to call back.
  A more robust system would retry with backoff.
